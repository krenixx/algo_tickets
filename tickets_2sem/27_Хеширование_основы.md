# Хэширование — основы

## Идея

Хотим хранить пары (ключ → значение) и отвечать на запросы поиска/вставки/удаления за $O(1)$ в среднем.

**Хеш-функция** $h(k)$ отображает ключ $k$ в индекс массива $[0, M)$. Массив называется **хеш-таблицей**.

```
ключ k  →  h(k)  →  ячейка таблицы
```

---

## Требования к хеш-функции

1. **Детерминированность:** $h(k)$ всегда одинаково для одного $k$.
2. **Равномерность:** ключи распределяются по ячейкам примерно поровну.
3. **Быстрота:** вычисляется за $O(1)$.

---

## Коллизия

Коллизия — когда $h(k_1) = h(k_2)$, но $k_1 \ne k_2$. Коллизии неизбежны (принцип Дирихле: если ключей больше, чем ячеек — хотя бы два попадут в одну).

Способы разрешения коллизий — два класса:
- **Цепочки (chaining / закрытая адресация)**
- **Открытая адресация**

---

## Хеш-функции

### Для целых чисел

**Метод деления:**
$$h(k) = k \bmod M$$
$M$ — простое, не близкое к степени двойки.

**Метод умножения:**
$$h(k) = \lfloor M \cdot \{k \cdot A\} \rfloor, \quad A \approx \frac{\sqrt{5}-1}{2} \approx 0.618$$
где $\{x\}$ — дробная часть. Менее чувствителен к выбору $M$.

### Для строк (полиномиальное хеширование)

$$h(s) = \left(\sum_{i=0}^{n-1} s[i] \cdot p^i\right) \bmod M$$

Обычно $p = 31$ или $p = 37$ (для строчных букв), $M$ — большое простое (например, $10^9 + 7$).

```cpp
long long poly_hash(const string& s, long long p = 31, long long mod = 1e9 + 7) {
    long long h = 0, pw = 1;
    for (char c : s) {
        h = (h + (c - 'a' + 1) * pw) % mod;
        pw = pw * p % mod;
    }
    return h;
}
```

**Двойное хеширование строк** (снижает вероятность коллизий):

```cpp
pair<long long, long long> double_hash(const string& s) {
    return {poly_hash(s, 31, 1e9 + 7), poly_hash(s, 37, 1e9 + 9)};
}
```

## Разрешение коллизий: цепочки

Каждая ячейка — список (цепочка) элементов с одним хешем.

```
h(k) = 2:   [ ] → [k1] → [k3] → null
h(k) = 5:   [ ] → [k2] → null
```

```cpp
struct HashMap {
    static const int M = 1e5 + 3;
    vector<pair<int,int>> table[M];   // table[h] — список (ключ, значение)

    int h(int k) { return ((k % M) + M) % M; }

    void insert(int k, int v) {
        int bucket = h(k);
        for (auto& [key, val] : table[bucket])
            if (key == k) { val = v; return; }
        table[bucket].push_back({k, v});
    }

    int* find(int k) {
        for (auto& [key, val] : table[h(k)])
            if (key == k) return &val;
        return nullptr;
    }

    void erase(int k) {
        auto& b = table[h(k)];
        b.erase(remove_if(b.begin(), b.end(),
            [k](auto& p){ return p.first == k; }), b.end());
    }
};
```

### Сложности (цепочки)

| Операция | Среднее | Худшее |
|----------|---------|--------|
| Поиск | $O(1 + \alpha)$ | $O(n)$ |
| Вставка | $O(1)$ | $O(n)$ |
| Удаление | $O(1 + \alpha)$ | $O(n)$ |

$\alpha = n / M$ — **коэффициент загрузки**. При $\alpha \le 1$ среднее время $O(1)$.

---

## Коэффициент загрузки и рехеширование

Когда $\alpha$ превышает порог (обычно $0.7$ для открытой адресации, $1.0$ для цепочек) — таблица **рехешируется**: создаём новую таблицу вдвое большего размера и перевставляем все элементы.

Амортизированная стоимость вставки остаётся $O(1)$.

---

## Универсальное хеширование

Чтобы защититься от специально подобранных входных данных (атака через коллизии), хеш-функцию выбирают **случайно** из семейства:

$$h_{a,b}(k) = ((ak + b) \bmod p) \bmod M$$

где $p$ — простое $> M$, $a \in [1, p-1]$, $b \in [0, p-1]$ — случайные. Это **универсальное семейство**: вероятность коллизии двух фиксированных ключей $\le 1/M$.

## Хеш-таблица в стандартной библиотеке C++

```cpp
#include <unordered_map>
#include <unordered_set>

unordered_map<string, int> mp;
mp["hello"] = 42;
mp.count("hello");   // 1 если есть
mp.erase("hello");

unordered_set<int> st;
st.insert(5);
st.count(5);         // 1 если есть
```

`unordered_map` / `unordered_set` — цепочки, $O(1)$ среднее. Используют `std::hash`.

**Осторожно:** `unordered_map` уязвим к хак-тестам (специально подобранные ключи → все в один бакет). Защита — своя хеш-функция:

```cpp
struct SafeHash {
    static uint64_t splitmix64(uint64_t x) {
        x += 0x9e3779b97f4a7c15; x = (x ^ (x >> 30)) * 0xbf58476d1ce4e5b9;
        x = (x ^ (x >> 27)) * 0x94d049bb133111eb; return x ^ (x >> 31);
    }
    size_t operator()(uint64_t x) const {
        static const uint64_t FIXED_RANDOM =
            chrono::steady_clock::now().time_since_epoch().count();
        return splitmix64(x + FIXED_RANDOM);
    }
};
unordered_map<long long, int, SafeHash> safe_map;
```

## Типичные вопросы

**В.** Почему $M$ должно быть простым?  
**О.** При $h(k) = k \bmod M$ простое $M$ равномернее распределяет ключи — меньше структурных совпадений при делении.

**В.** Что такое коэффициент загрузки $\alpha$?  
**О.** $\alpha = n/M$ — отношение числа элементов к числу ячеек. При $\alpha \to 1$ растёт число коллизий.

**В.** Чем опасен `unordered_map` на соревнованиях?  
**О.** Специально подобранные ключи могут дать все элементы в один бакет → $O(n)$ на запрос. Защита — `SafeHash` с рандомизацией.
