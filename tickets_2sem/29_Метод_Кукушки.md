# Метод кукушки (Cuckoo Hashing)

## Идея

Название — от птицы кукушки, которая выталкивает яйца из чужих гнёзд.

У каждого ключа **два возможных места** — $h_1(k)$ и $h_2(k)$ в двух разных таблицах (или двух половинах одной). Ключ **всегда** находится в одном из двух своих мест — поиск строго $O(1)$ в худшем случае.

При вставке, если оба места заняты — **выталкиваем** один из элементов на его альтернативное место, рекурсивно.

---

## Структура

```
Таблица T1 (размер M):   [ _ | k3 | _ | k1 | _ | ... ]
Таблица T2 (размер M):   [ k2 | _ | k1'| _ | k3'| ... ]
```

Инвариант: ключ $k$ лежит либо в `T1[h1(k)]`, либо в `T2[h2(k)]`.

---

## Поиск — $O(1)$ худший случай

```cpp
int* find(int k) {
    if (T1[h1(k)] == k) return &V1[h1(k)];
    if (T2[h2(k)] == k) return &V2[h2(k)];
    return nullptr;
}
```

Ровно два обращения к памяти — всегда.

---

## Удаление — $O(1)$

```cpp
bool erase(int k) {
    if (T1[h1(k)] == k) { T1[h1(k)] = EMPTY; return true; }
    if (T2[h2(k)] == k) { T2[h2(k)] = EMPTY; return true; }
    return false;
}
```

---

## Вставка — $O(1)$ амортизированно

### Алгоритм

1. Если `T1[h1(k)]` свободно — ставим туда, конец.
2. Если `T2[h2(k)]` свободно — ставим туда, конец.
3. Иначе — **выталкиваем** элемент из `T1[h1(k)]`, ставим $k$ на его место.
4. Выброшенный элемент пытаемся вставить в свою вторую ячейку. И так далее.
5. Если цикл не завершился за `MAX_ITER` шагов — **рехешируем** (меняем хеш-функции, перестраиваем).

```cpp
#include <vector>
#include <functional>
using namespace std;

struct CuckooMap {
    static const int EMPTY = INT_MIN;
    static const int MAX_ITER = 100;
    int M;
    vector<int> T1, T2, V1, V2;
    function<int(int)> h1, h2;

    CuckooMap(int M, function<int(int)> h1, function<int(int)> h2)
        : M(M), T1(M, EMPTY), T2(M, EMPTY), V1(M), V2(M), h1(h1), h2(h2) {}

    bool insert(int k, int v) {
        if (find(k)) { *find(k) = v; return true; }  // обновление
        int curK = k, curV = v;
        for (int iter = 0; iter < MAX_ITER; iter++) {
            // пробуем T1
            int i1 = h1(curK);
            if (T1[i1] == EMPTY) { T1[i1] = curK; V1[i1] = curV; return true; }
            swap(curK, T1[i1]); swap(curV, V1[i1]);  // выталкиваем из T1

            // пробуем T2
            int i2 = h2(curK);
            if (T2[i2] == EMPTY) { T2[i2] = curK; V2[i2] = curV; return true; }
            swap(curK, T2[i2]); swap(curV, V2[i2]);  // выталкиваем из T2
        }
        return false;  // цикл — нужно рехешировать
    }

    int* find(int k) {
        if (T1[h1(k)] == k) return &V1[h1(k)];
        if (T2[h2(k)] == k) return &V2[h2(k)];
        return nullptr;
    }

    bool erase(int k) {
        if (T1[h1(k)] == k) { T1[h1(k)] = EMPTY; return true; }
        if (T2[h2(k)] == k) { T2[h2(k)] = EMPTY; return true; }
        return false;
    }
};
```

---

## Граф кукушки

Визуализируем структуру как **двудольный граф**:
- Левые вершины — ячейки $T1$, правые — ячейки $T2$.
- Каждый ключ $k$ — ребро между $h_1(k)$ и $h_2(k)$.

```
h1(k1)=0 ——k1—— h2(k1)=1
h1(k2)=0 ——k2—— h2(k2)=3
h1(k3)=2 ——k3—— h2(k3)=1
```

**Вставка без цикла возможна тогда и только тогда, когда в этом графе нет цикла** (точнее, каждая связная компонента содержит не более одного цикла — тогда выталкивание всегда завершается).

При $\alpha \le 0.5$ граф кукушки случайный с малым числом рёбер — цикл появляется с малой вероятностью.

---

## Когда происходит рехеширование

Цикл в графе кукушки = зацикливание выталкиваний. Решение:

1. Взять новые случайные хеш-функции $h_1', h_2'$.
2. Перевставить все элементы в новую таблицу.

Вероятность цикла при $\alpha = n/(2M) < 0.5$ — $O(1/M)$ на одну вставку. Ожидаемое число рехешировний на $n$ вставок — $O(1)$. Итого вставка амортизированно $O(1)$.

---

## Варианты и расширения

### Три таблицы / три хеш-функции

Ключ имеет три возможных места. Максимальный $\alpha$ поднимается примерно до $0.91$.

### Cuckoo Hashing с корзинами (Buckets)

Каждая ячейка — маленький массив (размер $b = 4$ или $8$). Ищем любое свободное место в корзине. $\alpha$ поднимается до $\approx 0.98$.

## Сравнение с другими методами

| | Цепочки | Линейное | Двойное | Кукушка |
|---|---|---|---|---|
| Поиск (худший) | $O(n)$ | $O(n)$ | $O(n)$ | $O(1)$ ✓ |
| Поиск (среднее) | $O(1)$ | $O(1)$ | $O(1)$ | $O(1)$ |
| Вставка (среднее) | $O(1)$ | $O(1)$ | $O(1)$ | $O(1)$ аморт. |
| Удаление | просто | надгробия | надгробия | просто $O(1)$ |
| Макс. $\alpha$ | $>1$ | $<0.7$ | $<0.7$ | $<0.5$ |
| Кэш | плохо | отлично | плохо | хорошо (2 обращения) |

## Типичные вопросы

**В.** Почему поиск гарантированно $O(1)$?  
**О.** У ключа ровно два возможных места — больше проверять незачем.

**В.** Что делать при зацикливании вставки?  
**О.** Рехешировать: выбрать новые $h_1, h_2$ и перестроить таблицу.

**В.** Почему $\alpha < 0.5$?  
**О.** При большей загрузке граф кукушки с высокой вероятностью содержит цикл — вставки будут часто заканчиваться рехешированием.

**В.** В чём главное преимущество кукушки перед линейным зондированием?  
**О.** Поиск и удаление — строго $O(1)$ без зависимости от $\alpha$, тогда как линейное зондирование деградирует при высокой загрузке.

**В.** Как связан граф кукушки с корректностью вставки?  
**О.** Вставка без рехеширования возможна тогда и только тогда, когда в компоненте графа, куда добавляется новое ребро, нет цикла (компонента — дерево или пока ещё «псевдолес»).
